# Example dataset policy

Example datasets exist to test workflow mechanics, not to claim new science.

## Requirements

An example dataset should be:

- public or easy to regenerate;
- small enough for a laptop smoke test when possible;
- free of private, clinical, or sensitive data;
- documented with source, license, and citation;
- paired with expected artifacts;
- clear about what the example does not prove.

## Acceptable sources

Good candidates include:

- upstream tutorial data;
- public benchmark snippets;
- synthetic toy inputs;
- small files generated by a documented script;
- public-domain or clearly licensed sample data.

Avoid:

- private lab data;
- clinical or patient-linked data;
- unpublished collaborator data;
- large files that make CI or laptop tests impractical;
- examples that imply a new biological or materials discovery.

## Minimum metadata

Each example should document:

| Field | Meaning |
|---|---|
| `source` | URL, DOI, repository, or generation script |
| `license` | Data license or usage terms |
| `citation` | Citation if required |
| `size` | Approximate download or generated size |
| `inputs` | Input files and formats |
| `expected_artifacts` | Files the workflow should create |
| `limitations` | What the example cannot demonstrate |

## Rule for first demos

For early demos, prefer old, boring, well-understood public examples over impressive new cases. A stable reproducibility example is more useful than a flashy but hard-to-review output.
